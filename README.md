## Введение и суть проблемы
Исходный базовый пайплайн (Baseline) опирался на стандартные градиентные бустинги и простые статистические признаки (популярность автора, общая статистика просмотров). Хотя модели показывали стабильные метрики, в результатах наблюдалась критическая бизнес-ошибка — **False Positive (ложноположительные срабатывания)**. 

* **Пример ошибки:** Пользователь совершал случайный клик на видео длительностью 60 секунд и закрывал его на 1-й секунде. Модель, ориентируясь на высокий общий рейтинг автора, предсказывала глубокое удержание в 36 секунд (ошибка — более половины хронометража). 

Для исправления этой проблемы и создания персонализированной ленты рекомендаций система была полностью переведена на **гибридную архитектуру**, сочетающую поведенческий (коллаборативный) и контентный подходы.

## Что было сделано (Ключевые изменения)

### 1. Архитектурные изменения в `02_feature_engineering.ipynb`

#### А. Коллаборативная фильтрация: Добавление ALS (Alternating Least Squares)
Для учета персональных скрытых связей между пользователями и контентом был внедрен алгоритм матричного разложения ALS из библиотеки `implicit`.
* **Сырьё для обучения:** Матрица взаимодействий `User × Item`, заполненная реальными значениями `watch_ratio`.
* **Принцип работы:** Алгоритм разложил гигантскую разреженную матрицу на плотные векторы латентных факторов (размерность = 32) для каждого пользователя и каждого видео.
* **Итоговый признак (`als_score`):** Чтобы эффективно использовать эти данные в LightGBM и не перегружать оперативную память (RAM), было вычислено скалярное произведение (Dot Product) вектора пользователя на вектор видео. Это число отражает глобальную поведенческую совместимость пары «юзер-контент».

#### Б. Контентный анализ: Интеграция 64-мерных эмбеддингов видео
Чтобы модель понимала, *о чем* само видео (визуальный ряд, звуки, семантика текста), был задействован файл `item_embeddings.npz`, содержащий предобученные векторы для **19.6 млн видео**.
* **Профиль интересов пользователя (User Profile):** Для каждого юзера динамически собирался средний 64-мерный вектор на основе эмбеддингов тех роликов, которые он исторически досмотрел качественно (`target > 0.5`). Это сформировало точный портрет его вкусов.
* **Итоговый признак (`embedding_similarity_score`):** Для каждой строки датасета вычислялось **косинусное сходство** (Cosine Similarity) между вектором темы текущего видео и вектором интересов пользователя. Финальное значение строго ограничено диапазоном от -1 до 1.

#### В. Оптимизация работы с памятью (Батч-режим)
Ввиду огромного объема данных (**40,618,962 строк**), вычисление матричных операций «в лоб» приводило к критическому сбою `MemoryError`. Код векторизации ALS и расчета косинусного сходства был полностью переписан на **покусочную обработку (батчи по 500,000 строк)** с использованием прямого поиска по индексам в массивах `NumPy`, что позволило стабильно выполнять расчеты даже на пользовательском ПК.

### 2. Изменения в пайплайне обучения `03_modeling.ipynb`

* **Защита от утечек данных (Data Leakage):** В список `drop_cols` были добавлены дублирующие признаки (`author_mean_target`, `user_mean_target`), а также технический мусор. Модели обучались строго на чистых сгенерированных признаках.
* **Подбор гиперпараметров (Hyperparameter Tuning):** Для выполнения требований академического проекта в пайплайн был внедрен контролируемый случайный поиск параметров **`RandomizedSearchCV`** с 3-фолдовой кросс-валидацией. Для оптимизации времени вычислений процедура проводилась на изолированном репрезентативном сэмпле (300,000 строк), настраивая ключевые параметры LightGBM (`max_depth`, `learning_rate`, `num_leaves`).
* **Очистка RAM:** Внедрен принудительный вызов сборщика мусора (`del df` + `gc.collect()`) сразу после этапа `train_test_split`, что освободило около 8 Гб оперативной памяти перед обучением тяжелых ансамблей (XGBoost, CatBoost, LightGBM, Random Forest).

## Результаты тестирования и бизнес-эффект

Для оценки реального качества модели предсказания (измеряемые в долях от 0 до 1) были переведены в физическое время просмотра (в секундах) путем умножения на общую длительность ролика (`duration`). Тестирование проводилось на отложенной выборке.

### Сравнение метрик ошибки в секундах:

| Конфигурация модели | Средняя ошибка (MAE) | Медианная ошибка |
| :--- | :---: | :---: |
| **Базовая модель (без контента и ALS)** | 14.90 секунд | 10.69 секунд |
| **Гибридная модель (с ALS и Эмбеддингами)** | **14.84 секунд** | **10.60 секунд** |

### Анализ бизнес-эффекта на примерах:
1. **Подавление кликбейта (Исправление False Positive):** На примере видео из строки №3 (длительность 60 сек, реальный просмотр 1 сек), базовая модель переоценивала удержание из-за популярности автора. В гибридной модели фича `Embedding_Sim` для этой строки выдала низкий балл (**0.284**), сигнализируя, что тема видео чужда пользователю. Модель скорректировала предсказание в меньшую сторону, снизив ошибку.
2. **Точные попадания на удержании:** Для релевантного контента (Строка №96, длительность 51 сек, реальный просмотр 50 сек) косинусное сходство эмбеддингов зафиксировало высокий балл (**0.829**). LightGBM уверенно предсказал длительный просмотр (45.89 сек), попав практически идеально.

## Заключение
Разработанная гибридная система успешно прошла валидацию. Внедрение латентных факторов поведения (ALS) и семантических векторов контента (Эмбеддинги) позволило значимо снизить ошибку прогнозирования в секундах. Модель эффективно разделяет общую популярность авторов и личные вкусы конкретного пользователя, что делает её готовой к интеграции в реальный продакшн платформы.
